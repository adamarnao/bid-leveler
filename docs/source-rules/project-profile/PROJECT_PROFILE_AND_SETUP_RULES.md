# Project Profile And Setup Rules

## Purpose

Project Setup is a preconstruction profiling tool. Its purpose is to collect enough project information to classify the work, generate and review trade bid packages, prepare comprehensive ITBs, and support downstream estimating workflows.

Project Setup is used to:

- classify the project
- generate trade and bid packages
- trigger context-specific scope requirements
- prepare ITBs
- support historical pricing
- support estimate review
- support proposal and clarification output

Project Setup is not just an administrative project record. It is the project profile that drives package creation, invite readiness, bid collection, leveling, financial review, and proposal preparation.

## Project Profile Model

Project Profile is the parent source-of-truth for setup. It includes:

- classification
- global attributes
- logistics
- procurement
- package default assumptions
- pricing metrics

Classification is one section inside Project Profile. The canonical classification source rules live in `../project-classification/` and define sector, subsector / facility type, work type, and context tags.

## Setup Flow Stages

The Project Setup / Project Profile workflow should follow these stages:

1. Project Intake
2. Initial Plan Review / Project Profile
3. Global Conditions / Constraints
4. Suggested Trade Packages
5. Package Review
6. Subcontractor Invite List
7. ITB Review
8. Send ITBs

### 1. Project Intake

Project Intake creates the project record and captures the minimum identifying information needed to start profiling the pursuit.

Typical intake data includes project name, client or bid solicitor, market location, estimator or internal bid contact, bid dates, and document access instructions.

### 2. Initial Plan Review / Project Profile

Initial Plan Review turns the project record into a Project Profile. Its classification section identifies sector, subsector / facility type, work type, and context tags. Its global attributes identify CSI version, approximate size, major building facts, and scope flags that influence package generation.

### 3. Global Conditions / Constraints

Global Conditions / Constraints captures project-level constraints that affect multiple packages or the ITB as a whole. These include procurement requirements, site logistics, occupied operations, public/private bid status, wage requirements, bond requirements, tax status, OFCI/CFCI/OFOI/NIC assumptions, site walk status, and similar conditions.

### 4. Suggested Trade Packages

Suggested Trade Packages are generated from the Project Profile, selected CSI tags, classification, package default assumptions, and trade taxonomy rules. Bid Packages are the user-facing scope units. CSI tags are supporting classification and scope clarity tags inside those packages.

### 5. Package Review

Package Review is where estimators confirm which packages are included, excluded, split, combined, renamed, clarified, or adjusted before ITBs are sent.

Each package must be reviewed before ITB launch.

### 6. Subcontractor Invite List

The Subcontractor Invite List is generated by matching bid packages and their CSI tags to subcontractor coverage, contacts, and package-specific recipient rules. The estimator reviews recipients by package before launch.

### 7. ITB Review

ITB Review confirms that the project-level facts, package-level scope, required forms, requested alternates, requested allowances, requested unit prices, documents, instructions, and recipients are ready to send.

### 8. Send ITBs

Send ITBs launches invitations by package and recipient. Sending may support package-level sends, recipient-specific sends, resends, and later event-specific invitations.

## Required Before ITB At Project Level

The following project-level information is required or conditionally required before ITBs can be sent:

- project name
- address / market location
- client / owner / bid solicitor
- estimator / internal bid contact
- bid due dates
- document access instructions
- project CSI version
- sector
- subsector / facility type
- work type
- context tags
- square footage
- floors / stories
- building condition
- structure type if relevant
- occupied / active operations status
- sitework scope flag
- exterior envelope scope flag
- site walk status: scheduled / TBD / not applicable
- public/private bid
- prevailing wage if applicable
- bond requirements if applicable
- tax status if applicable
- OFCI / CFCI / OFOI / NIC assumptions if known

These requirements define ITB readiness. They do not mean the full project setup is complete for estimating, leveling, proposal generation, accounting, or historical reporting.

## Required Per Package Before ITB

Each bid package must be reviewed before ITBs are sent for that package.

Package review includes:

- include / exclude decision
- split / combine decision
- package name
- trade / specialization
- CSI tags
- CSI tag roles: core / optional / possible / excluded
- scope summary
- package-specific finish level
- package-specific work intensity
- package-specific demo intensity if relevant
- package-specific MEP coordination intensity if relevant
- package-specific low voltage / controls / specialty system intensity if relevant
- clarifications
- exclusions
- requested alternates
- requested allowances
- requested unit prices
- required attachments/forms
- subcontractor recipient review status

Package review is the estimator's opportunity to convert generated package suggestions into real ITB-ready scope definitions.

## Global Vs Package-Specific Rules

Project Setup collects global project facts, classification, site constraints, logistics constraints, procurement constraints, pricing metrics, and package-generation triggers.

Package Review collects trade-specific scope decisions, finish level, work intensity, exclusions, clarifications, alternates, allowances, and unit prices.

Project-level finish, demo, MEP, low voltage, controls, and specialty-system intensity may exist as default assumptions. Those assumptions are not final truth for every package.

The final package-specific truth lives on the reviewed bid package.

### Project-Level Responsibilities

Project-level setup should own:

- project identity and location
- client / owner / bid solicitor
- internal bid contact
- project classification
- CSI version
- document access instructions
- bid due dates
- public/private bid status
- wage, bond, tax, and procurement requirements
- global site and logistics constraints
- broad scope flags such as sitework or exterior envelope presence
- default assumptions used to suggest packages
- pricing metrics used for historical comparisons, estimate review, and reporting

### Package-Level Responsibilities

Package review should own:

- whether the package is included in the ITB set
- whether generated scopes should be split or combined
- package name and trade / specialization
- CSI tags and CSI tag roles
- package-specific scope summary
- package-specific finish and work intensity
- package-specific demo, MEP, low voltage, controls, and specialty-system coordination intensity
- package clarifications
- package exclusions
- package alternates, allowances, and unit prices requested from bidders
- package-specific attachments and forms
- package recipient review status

## Source-Of-Truth Rule

Project Profile facts generate package suggestions. Reviewed Bid Packages are the user-facing scope units used for ITBs, bid collection, bid leveling, estimate review, and proposal preparation.

CSI codes are technical tags used for classification, scope clarity, matching, leveling, reporting, and cross-version compatibility. They are not the primary user-facing deliverable.
